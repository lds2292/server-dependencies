# i18n (한국어/영어 로컬라이징) 구현 기획서

## 개요

모든 UI 텍스트와 서버 에러 메시지에 하드코딩된 한국어를 `vue-i18n` 기반의 번역 시스템으로 전환한다.
한국어(ko)와 영어(en) 두 언어를 지원하며, 브라우저 언어 자동 감지 + 수동 전환 + localStorage 저장을 제공한다.

---

## 1. 현황 분석

### 1.1 현재 문제점

- **클라이언트**: 10개 뷰, 17개 컴포넌트에 한국어 문자열이 직접 하드코딩되어 있다.
- **서버**: 에러 메시지가 한국어로 하드코딩되어 있으나, 대부분 `code` 필드를 함께 반환한다. 일부 컨트롤러에서는 `code` 없이 `error` 메시지만 반환하는 경우가 있다.
- **i18n 라이브러리**: 미설치 상태. `main.ts`에서 Pinia + Router만 사용 중.

### 1.2 서버 에러 코드 현황

서버는 이미 대부분의 에러에 `code` 필드를 포함하고 있다:
- `authService.ts`: `EMAIL_TAKEN`, `USERNAME_TAKEN`, `INVALID_CREDENTIALS`, `OAUTH_ONLY_ACCOUNT`, `SESSION_EXPIRED`, `VALIDATION_ERROR`, `UNSUPPORTED_PROVIDER`, `INVALID_GOOGLE_TOKEN`
- `projectService.ts`: `NOT_FOUND`, `ACCESS_DENIED`, `MEMBER_NOT_FOUND`, `ALREADY_MEMBER`
- `invitationService.ts`: `NOT_FOUND`, `ACCESS_DENIED`, `ALREADY_MEMBER`, `ALREADY_INVITED`, `ALREADY_PROCESSED`
- `googleAuthService.ts`: `INVALID_GOOGLE_TOKEN`, `EMAIL_NOT_VERIFIED`

**코드가 없는 에러** (변경 필요):
- `server/src/middleware/authenticate.ts`: `error`만 반환 (코드 없음)
- `server/src/controllers/authController.ts`: 일부 400 에러에 코드 없음
- `server/src/controllers/projectController.ts`: 일부 400/404/500 에러에 코드 없음
- `server/src/controllers/invitationController.ts`: 500 에러에 코드 없음
- `server/src/controllers/graphController.ts`: 500 에러에 코드 없음

## 2. 기술 설계

### 2.1 라이브러리 설치

```bash
npm install vue-i18n@10 --workspace=client
```

### 2.2 vue-i18n 설정

**신규 파일: `client/src/i18n/index.ts`**

```typescript
import { createI18n } from 'vue-i18n'
import ko from './locales/ko'
import en from './locales/en'

function detectLocale(): 'ko' | 'en' {
  const saved = localStorage.getItem('app-locale')
  if (saved === 'ko' || saved === 'en') return saved
  const browserLang = navigator.language || (navigator as any).userLanguage || ''
  return browserLang.startsWith('ko') ? 'ko' : 'en'
}

const i18n = createI18n({
  legacy: false,          // Composition API 모드
  locale: detectLocale(),
  fallbackLocale: 'ko',   // 번역 누락 시 한국어로 폴백
  messages: { ko, en },
})

export default i18n

/** 언어 변경 및 localStorage 저장 */
export function setLocale(locale: 'ko' | 'en'): void {
  i18n.global.locale.value = locale
  localStorage.setItem('app-locale', locale)
  document.documentElement.lang = locale
}

export function getLocale(): 'ko' | 'en' {
  return i18n.global.locale.value as 'ko' | 'en'
}
```

**main.ts 변경:**

```typescript
import { createApp } from 'vue'
import './style.css'
import { createPinia } from 'pinia'
import router from './router'
import i18n from './i18n'       // 추가
import App from './App.vue'

const app = createApp(App)
app.use(createPinia())
app.use(i18n)                   // 추가
app.use(router)
app.mount('#app')
```

### 2.3 번역 파일 구조

```
client/src/i18n/
  index.ts                    # vue-i18n 인스턴스 생성 + setLocale/getLocale
  locales/
    ko.ts                     # 한국어 번역 (기본값)
    en.ts                     # 영어 번역
```

각 번역 파일은 페이지/기능 단위로 네스트된 객체 구조를 사용한다:

```typescript
// ko.ts 구조 예시
export default {
  common: { ... },
  auth: { ... },
  projects: { ... },
  project: { ... },
  settings: { ... },
  account: { ... },
  audit: { ... },
  graph: { ... },
  nodes: { ... },
  modals: { ... },
  errors: { ... },
}
```

### 2.4 키 네이밍 컨벤션

| 규칙 | 예시 |
|------|------|
| 최상위: 페이지 또는 기능 영역 | `auth`, `projects`, `graph`, `common` |
| 2단계: 컴포넌트 또는 섹션 | `auth.login`, `graph.canvas`, `modals.server` |
| 3단계: 구체적 텍스트 | `auth.login.title`, `auth.login.submit` |
| 동적 값: 중괄호 보간 | `'{count}건'` -> `'{count} items'` |
| 복수형: 미사용 | 한국어에는 복수형이 없으므로 `{count}`로 통일 |

### 4.1 방침

### 4.4 클라이언트 에러 처리 유틸리티

**신규 파일: `client/src/utils/errorMessage.ts`**

```typescript
import { useI18n } from 'vue-i18n'

/**
 * API 에러 응답에서 번역된 에러 메시지를 추출한다.
 * code 필드가 있으면 errors 네임스페이스에서 번역, 없으면 fallback 반환.
 */
export function getErrorMessage(
  err: unknown,
  t: ReturnType<typeof useI18n>['t'],
  fallbackKey: string = 'common.serverError',
): string {
  const e = err as { response?: { data?: { code?: string; error?: string } } }
  const code = e.response?.data?.code
  if (code) {
    const key = `errors.${code}`
    const translated = t(key)
    // vue-i18n은 키가 없으면 키 자체를 반환하므로 비교하여 폴백 처리
    if (translated !== key) return translated
  }
  return t(fallbackKey)
}
```

## 9. 제약 조건 및 주의사항

1. **CLAUDE.md 규칙 준수**
   - 이모지 사용 금지 -- `ImpactPanel.vue`의 방화벽 아이콘은 이모지(`🔒`, `📋`)를 사용 중이다. i18n 적용 시 이를 SVG 아이콘이나 텍스트로 교체하는 것은 이 기획의 범위 밖이지만, 추후 별도로 처리해야 한다.
   - UI 색상 하드코딩 금지 -- 새로 추가하는 `.locale-badge` 스타일은 반드시 CSS 변수를 사용한다.

2. **기존 패턴 준수**
   - `GraphCanvas.vue` 등 `<script setup>` 내에서 `$t()` 대신 `useI18n()`의 `t()` 함수를 사용한다.
   - 템플릿에서는 `{{ $t('key') }}` 또는 `v-bind`로 사용한다.

3. **Store에서의 i18n 사용**
   - Pinia store는 Vue 컴포넌트 밖이므로 `useI18n()`을 직접 사용할 수 없다.
   - Store에서 에러 메시지가 필요한 경우 i18n 키 문자열을 반환하고, 컴포넌트에서 `$t()`로 변환한다.
   - 예: `graph.ts`의 `saveError`는 `'store.graph.saveError'` 키를 저장하고, `ProjectView.vue`에서 `$t(store.saveError)`로 표시한다.

4. **번역 파일 크기**
   - 한 파일에 모든 키를 넣되, 주석으로 섹션을 구분한다.
   - 추후 파일이 커지면 `ko/auth.ts`, `ko/graph.ts` 등으로 분리할 수 있다.

5. **fallbackLocale: 'ko'**
   - 영어 번역이 누락되면 한국어가 표시된다. 점진적 영어 번역이 가능하다.

6. **HTML 엔티티 및 특수 문자**
   - `{{ $t() }}` 내에서 HTML이 필요한 경우 `v-html`과 `$t()` 조합 대신, 가능하면 구조를 분리하여 순수 텍스트만 번역한다.

7. **서버 에러 메시지 하위 호환**
   - 서버의 `error` 필드는 영어 메시지로 변경하되, 클라이언트는 항상 `code`를 우선 참조한다.
   - `code`가 없거나 매핑되지 않는 경우 `common.serverError` 폴백을 표시한다.
