## 1. 현재 노드 구조 분석

### 1.1 물리적 사양

| 항목 | 값 |
|------|------|
| 렌더링 방식 | SVG foreignObject 없이 순수 SVG (rect + text + line + circle) |
| 고정 크기 | 187 x 74 px |
| 모서리 반경 | rx="6" |
| 좌표 기준점 | 노드 중앙 (translate 기반, x=-86 ~ x=101, y=-37 ~ y=37) |

### 1.2 내부 레이아웃

```
+--------+---------------------------+
|  ICON  |  name (node-label)        |
|  26px  |  sub info (node-sub)      |
| column |  meta info (node-meta)    |
+--------+---------------------------+
  x:-86    x:-60                 x:101
  ~-60     separator line
```

- **좌측 아이콘 컬럼** (26px 너비): 노드 타입을 나타내는 SVG 아이콘. infra/dns 타입은 벤더별 약어 배지(PG, MY, RD, R53, CF 등)가 원형 안에 표시됨
- **구분선**: x=-60 위치에 수직 구분선
- **우측 텍스트 영역**: 3행 구조
  - 1행 (dy=-9): 노드 이름 -- `node-label` (11px, bold 700, white)
  - 2행 (dy=6): 부가 정보 -- `node-sub` (9px, bold 600, 0.65 opacity white)
  - 3행 (dy=19): 메타 정보 -- `node-meta` (9.5px, 0.5 opacity white)
- 텍스트는 모두 `text-anchor="middle"`, x=21 (텍스트 영역의 수평 중앙)

### 1.3 노드 타입별 정보 구성

| 타입 | 배경색 | 1행 (label) | 2행 (sub) | 3행 (meta) |
|------|--------|-------------|-----------|------------|
| Server | 어두운 앰버 (`--accent-bg-medium`) | 서버명 | 첫 번째 내부IP (+N) | NAT: 첫 번째 NAT IP (+N) |
| L7 | 어두운 보라 (`--node-l7-bg-deep`) | LB명 | IP / NAT IP (없으면 "L7 Load Balancer") | N개 서버 |
| Infra | 밝은 배경 (`--node-infra-bg-light`) | 인프라명 | 인프라 타입 (e.g., PostgreSQL) | host:port |
| External | 어두운 초록 (`--node-ext-bg-deep`) | 서비스명 | "외부 서비스" (고정 텍스트) | 담당자 N명 |
| DNS | 밝은 배경 (`--node-dns-bg-light`) | 도메인명 | DNS 타입 (e.g., A, CNAME) | 레코드 값 |

### 1.4 상태 표현

| 상태 | 시각적 피드백 |
|------|-------------|
| 선택됨 | 앰버 glow 필터 + 타입별 stroke 색상 (3px) |
| hover | 타입별 glow 필터 (purple/cyan/green) |
| 영향 받음 (impacted) | 빨간 glow 필터 |
| 경로 탐색 중 | 비경로 노드 opacity 0.15, 경로 노드 앰버 glow |
| 순환 의존성 | 빨간 glow + 우상단 빨간 원형 "!" 배지 |
| 연결 소스 | brightness(1.3) 필터 |
| 연결 타겟 | brightness(1.5) 필터 |
| 다중 선택 | 주황 marching ants 애니메이션 (stroke-dasharray) |

### 1.5 인터랙션

- 클릭: 노드 선택, 영향도 분석 트리거
- 드래그: 노드 이동 (D3 force simulation 기반)
- Ctrl+드래그: 의존성 연결 (화살표 미리보기)
- 우클릭: 컨텍스트 메뉴 (수정/삭제/의존성 추가/경로 탐색)
- hover: 타입별 glow 피드백

## 2. 독창성 평가

### 2.1 차별화된 요소

**(A) 좌측 벤더 아이콘 패널 -- 양호**

infra/dns 노드에서 좌측 컬럼에 벤더별 약어 배지(PG, MY, RD 등)를 원형으로 표시하고, 해당 영역의 배경색을 벤더 고유 색상으로 칠하는 방식은 일반적인 노드 에디터에서 거의 볼 수 없는 패턴이다. Node-RED나 n8n의 아이콘이 단순히 브랜드 로고를 보여주는 것과 달리, 축약어 기반의 컬러 배지는 더 정보 밀도가 높고 일관된 시각 언어를 제공한다.

**(B) 밝은/어두운 배경 분리 전략 -- 양호**

infra와 dns 노드가 밝은 배경(`--node-infra-bg-light`, `--node-dns-bg-light`)을 사용하고, 나머지가 어두운 배경을 사용하는 것은 다크모드 캔버스에서 의도적으로 "이질적인" 시각 신호를 만든다. 인프라 노드가 "물리적으로 다른 계층"에 있다는 것을 배경색의 명도 차이로 암시하는 것은 좋은 디자인 판단이다.

**(C) 의존성 라인의 다층 시각화 -- 양호**

일반적인 노드 에디터는 연결선을 단일 색상으로 표현하지만, 이 프로젝트는 출발-도착 노드 색상의 그라디언트(`linearGradient`), 영향도 분석(빨간 대시), 의존 경로(초록 대시), 경로 탐색(앰버 대시), 방화벽 아이콘 등 여러 상태를 레이어로 표현한다. 이 정도의 라인 상태 분화는 경쟁 제품에서 보기 어렵다.

**(D) 순환 의존성 실시간 감지 및 시각화 -- 독창적**

노드에 순환 의존성이 발생하면 빨간 glow + 경고 배지("!")를 실시간으로 표시하고, 상단 배너로 감지된 노드 수를 알려주는 것은 운영 도구로서 실용적인 차별점이다. Node-RED, n8n 등은 실행 시점에서만 순환을 감지하지만, 이 프로젝트는 편집 중에 즉시 피드백을 제공한다.

### 2.2 일반적인 패턴과 유사한 요소

**(A) 고정 크기 직사각형 노드**

모든 노드가 동일한 187x74px 크기를 사용한다. 이는 n8n, Node-RED와 유사한 접근이며, Figma의 가변 크기 프레임이나 Blender의 소켓 기반 노드와는 구별된다. "나쁜 선택"은 아니지만 독창적이지도 않다.

**(B) text-anchor: middle 중앙 정렬**

모든 텍스트가 수평 중앙 정렬이다. 이는 SVG 노드에서 가장 흔한 패턴이지만, 긴 텍스트가 양쪽으로 잘리는 문제가 있고, 좌측 정렬이 가독성에서 일반적으로 우월하다.

**(C) 동일한 3행 텍스트 구조**

5개 노드 타입 모두 동일한 3행 레이아웃을 공유한다. 정보 종류가 다름에도 동일한 구조를 강제하므로, 특정 타입에서 정보 활용이 비효율적이다 (아래 3.2에서 상세).

## 3. 정보 노출 관련 문제점

### 3.1 External 노드의 정보 낭비

현재 External 노드의 2행은 항상 "외부 서비스"라는 고정 텍스트를 표시한다. 이것은 이미 노드 색상/아이콘으로 전달하고 있는 정보를 텍스트로 중복하는 것이며, 귀중한 공간을 낭비한다. "담당자 N명"이라는 3행 정보 역시 대부분의 사용 맥락에서 즉시 필요한 정보가 아니다.

**제안**: 2행에 방화벽 유무 또는 화이트리스트 상태("FW Required" / "Whitelist")를 표시하면 운영 관점에서 더 유용하다. 3행에는 담당자 이름(1명이면 이름 직접 표시, 2명 이상이면 "홍길동 외 1명")을 표시하는 것이 낫다.

### 3.2 Server 노드의 IP 가독성

Server 노드의 2행에 내부 IP를 표시하지만, IP 주소는 노드의 9px 폰트 크기에서 읽기 어렵다. 또한 `+N` 표기가 뒤에 붙지만, 사용자가 실제로 필요한 것은 "이 서버가 어떤 팀 소속인지"일 가능성이 높다.

**제안**: 2행에 팀명을 표시하고, 3행에 IP(첫 번째만)를 mono 폰트로 표시하는 것이 정보 위계에 맞다. IP 상세는 사이드 패널(ImpactPanel)에서 이미 제공하므로, 노드에서는 식별에 충분한 최소 정보만 보여주면 된다.

### 3.3 L7 노드의 조건부 텍스트 복잡성

L7 노드의 2행은 `ip || natIp || 'L7 Load Balancer'`라는 3단 폴백 로직이 있다. IP가 있으면 IP를 보여주고, 없으면 타입 라벨을 보여주는 것은 "같은 행이 맥락에 따라 완전히 다른 정보를 보여준다"는 의미이므로, 사용자가 L7 노드를 훑어볼 때 일관된 패턴을 기대할 수 없다.

**제안**: 2행은 항상 멤버 서버 수를 표시하고("3 servers"), 3행에 IP 정보를 배치하면 일관성이 확보된다.

### 3.4 텍스트 잘림 처리의 한계

`truncate(text, 20)`으로 20자 초과 시 말줄임을 적용하지만, SVG text 요소는 CSS `text-overflow: ellipsis`를 지원하지 않으므로 실제 렌더링 너비와 무관하게 글자 수로만 잘린다. 한글 20자와 영문 20자의 실제 너비 차이가 크기 때문에, 한글 이름은 노드 밖으로 넘칠 수 있고 영문 이름은 불필요하게 일찍 잘릴 수 있다.

**제안**: 영문/한글 혼합을 고려한 바이트 기반 잘림이나, SVG `textLength` 속성 활용을 검토할 수 있다. 또는 실제 텍스트 렌더링 너비를 `getComputedTextLength()`로 측정하여 동적으로 잘림 위치를 결정하는 방법도 있다.

### 3.5 노드 크기의 고정성

모든 노드가 동일한 187x74px이므로, 이름이 매우 짧은 노드(e.g., "DB1")와 매우 긴 노드(e.g., "production-api-gateway-v2")가 같은 크기를 차지한다. 이것은 캔버스의 공간 효율성을 떨어뜨린다.

다만, 가변 크기 노드는 D3 force simulation에서 충돌 감지가 복잡해지고, 연결선 계산도 어려워지므로, 현재 단계에서 고정 크기를 유지하는 것은 합리적인 트레이드오프이다. 장기적으로는 2~3단계(소/중/대)의 노드 크기를 지원하는 것을 고려할 수 있다.

## 4. 경쟁 제품 대비 비교

### 4.1 Node-RED / n8n 대비

| 관점 | Node-RED / n8n | Server Dependencies |
|------|---------------|-------------------|
| 노드 목적 | 워크플로우 "단계" 표현 | 인프라 "엔티티" 표현 |
| 포트/소켓 | 입력/출력 포트가 노드 좌우에 명시적으로 존재 | 포트 없음, 노드 전체가 연결 대상 |
| 연결 방식 | 포트 to 포트 드래그 | Ctrl+드래그로 노드 to 노드 |
| 정보 밀도 | 낮음 (이름 + 아이콘 정도) | 중간 (3행 텍스트 + 아이콘) |
| 상태 표현 | 실행 상태 (성공/실패) | 영향도/순환/경로 |

**차별점**: Server Dependencies는 "실행"이 아닌 "관계"를 시각화하는 도구이므로, 포트 기반 연결 모델 대신 노드 간 직접 연결 모델을 사용하는 것은 적절하다.

### 4.2 Figma / Miro 대비

| 관점 | Figma / Miro | Server Dependencies |
|------|-------------|-------------------|
| 노드 크기 | 자유 크기 | 고정 크기 |
| 중첩/그룹 | 프레임 안에 프레임 가능 | L7 멤버 관계가 시각적으로는 분리됨 |
| 텍스트 편집 | 인라인 편집 | 모달을 통한 편집 |

**차별점**: L7 노드와 멤버 서버 간의 관계를 점선 연결로 표현하는 것은 Figma의 "Auto Layout"과 다른 접근이다. 그러나 L7의 멤버 서버가 캔버스에서 물리적으로 분산되어 있으므로, "이 서버들이 같은 LB 뒤에 있다"는 관계가 즉각적으로 인지되기 어렵다.

### 4.3 Blender Node Editor 대비

| 관점 | Blender | Server Dependencies |
|------|---------|-------------------|
| 노드 구조 | 가변 높이, 속성 행이 동적으로 추가됨 | 고정 3행 |
| 소켓 | 타입별 색상 코딩된 입출력 소켓 | 없음 |
| 연결선 | 베지어 커브 | 직선 |

Server Dependencies의 직선 연결은 블렌더의 커브보다 "다이어그램"에 가깝고, 이는 인프라 의존성 문서화 도구라는 성격에 맞다.

## 5. 개선 아이디어

### 5.1 노드 호버 시 확장 정보 툴팁 (단기)

현재 hover 시 glow 효과만 있고, 추가 정보를 확인하려면 클릭하여 사이드 패널을 열어야 한다. hover 시 노드 근처에 가벼운 SVG 툴팁(또는 foreignObject 기반)으로 상세 정보를 보여주면, 탐색 속도가 크게 향상된다.

표시할 정보 예시:
- Server: 팀, 전체 IP 목록, 설명 첫 줄
- Infra: host:port, 설명
- External: 방화벽 상태, 담당자 목록
- DNS: TTL, 전체 레코드 값

### 5.2 연결 포트 힌트 (단기)

현재 노드 전체가 연결 대상이므로 Ctrl+드래그 시 어디에서 선이 나가는지 예측이 어렵다. 노드 hover 시 우측 중앙에 작은 연결 핸들(예: 6px 원)을 표시하고, 그 핸들에서 드래그하면 연결이 시작되도록 하면, "Ctrl+드래그" 조합 없이도 직관적으로 연결을 만들 수 있다.

### 5.3 L7 그룹 시각 영역 (중기)

L7 로드밸런서와 그 멤버 서버들을 반투명 영역(convex hull 또는 단순 bounding box)으로 감싸면, 물리적으로 떨어진 서버들이 같은 LB에 속한다는 것을 훨씬 빠르게 인지할 수 있다. 현재의 점선 연결만으로는 복잡한 그래프에서 멤버십이 시각적으로 묻힌다.

구현 참고: D3의 `d3.polygonHull()`로 멤버 좌표의 볼록 껍질을 계산하고, 패딩을 적용한 path를 렌더링할 수 있다.

### 5.4 노드 텍스트의 좌측 정렬 전환 (단기)

현재 `text-anchor="middle"`로 모든 텍스트가 중앙 정렬이지만, 노드 이름과 부가 정보는 좌측 정렬이 가독성에서 우월하다. 좌측 정렬로 전환하면:
- 이름의 시작 위치가 일정하여 빠른 스캔이 가능
- 긴 텍스트의 잘림이 오른쪽에서만 발생하여 예측 가능
- text-anchor를 "start"로, x를 구분선 우측(예: -52)으로 변경하면 됨

### 5.5 정보 행별 역할 재정의 (중기)

현재 3행의 역할이 노드 타입마다 다르고 일관성이 부족하다. 아래와 같은 규칙을 적용하면 사용자가 패턴을 학습하기 쉬워진다:

| 행 | 역할 | 예시 |
|----|------|------|
| 1행 | 고유 이름 | "auth-api", "prod-db-01" |
| 2행 | 핵심 속성 (타입에 따라 다름) | "Backend팀", "PostgreSQL", "CNAME", "FW Required" |
| 3행 | 네트워크/기술 식별자 | "10.0.1.5", "db.internal:5432", "cdn.example.com" |

이 규칙을 적용하면:
- Server: 이름 / 팀명 / 첫 번째 IP
- L7: 이름 / N개 서버 / IP
- Infra: 이름 / 인프라 타입 / host:port (현재와 동일, 이미 양호)
- External: 이름 / 방화벽 상태 또는 담당자 / URL 또는 설명 첫 줄
- DNS: 이름 / 레코드 타입 / 레코드 값 (현재와 동일, 이미 양호)

### 5.6 연결선을 직선에서 부드러운 커브로 (장기)

현재 직선 연결은 노드가 밀집한 영역에서 선이 겹쳐 판독이 어렵다. 완만한 베지어 커브(quadratic 또는 cubic)를 사용하면 선이 자연스럽게 분산되어 가독성이 향상된다. 다만 이는 렌더링 성능에 영향을 줄 수 있으므로 노드 수가 많은 경우에 대한 성능 테스트가 필요하다.
